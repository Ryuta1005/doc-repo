- Spec: -

## 目的

doc-repo の中心価値を「static documentation site generator」から、Git と Markdown を正本にしたブラウザベースのドキュメントワークスペースへ整理する。既存の静的生成は 019 時点では互換機能として扱い、プロダクト説明・CLI契約・テスト・ロードマップ上の正規入口を `serve` ベースの React + Hono ワークスペースへ寄せる。

## 受け入れ条件

- プロダクト説明で、doc-repo を静的サイト生成ツールではなく、Markdown ドキュメントをブラウザ上で閲覧・編集するドキュメント管理ツールとして表現できている
- `serve` が React Viewer の正規入口であることを README、product、roadmap、architecture、backlog のいずれか必要箇所へ反映できている
- 設定ファイルや CLI の `port` は Front/Back に分かれず、Hono が React UI・HTTP API・SSE・静的アセットを同一 port / 同一 origin で配信する前提になっている
- `doc-repo [scopePath]` による静的生成を残す場合、その位置づけが「中核価値」ではなく互換機能または移行中の出力手段として整理されている
- 静的生成を廃止・非推奨・互換維持のいずれにするかを Spec または research で決定している
- 旧テンプレートUI（`templates/page.html`, `templates/styles.css`, `templates/app.js`）を残す場合、残す理由と削除条件が明確になっている
- 静的生成中心のテストが、正規機能の回帰テストなのか互換維持テストなのか分類されている
- 010 の編集・保存機能が、静的生成前提ではなく `serve` の React + Hono ワークスペース前提で設計開始できる

## 技術検証

- 静的生成を完全廃止する場合の影響範囲を確認する
  - README/package description
  - CLI contract
  - `doc-repo [scopePath]` の既存テスト
  - `templates/` と `core/site/renderPages`
  - オフライン閲覧に関する既存仕様
- 静的生成を React Viewer と統合する場合は、Hono API がない環境で文書一覧・本文HTMLを供給する方式を検証する
  - 生成済み JSON
  - HTML への埋め込みデータ
  - React Viewer の data provider 分離
- 019 の完了条件を壊さず、010 の保存 API / 編集 UI 設計に不要な静的生成前提が入り込まないことを確認する
