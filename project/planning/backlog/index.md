# Backlog Index

このディレクトリは、Issue 管理の単一ソースとして使う。

## 参照元（正本）

- 運用ルールの正本: [../issue-workflow.md](../issue-workflow.md)
- ユーザー価値と受け入れ条件の正本: [../story-map.md](../story-map.md)

## 使い方

- 一覧はこの `index.md` で管理する
- チケット番号は全種別で共通の連番を使う（001, 002, 003...）
- `001_エピック` と `001_ストーリー` のような番号重複は不可
- ファイルは原則 Story または技術タスク（機能に直接ひも付かない Task/Spike）を作成する
- Epic はファイルではなくフォルダで管理する
- 種別は Epic, Story, Task, Spike のいずれか
- ステータスは Todo, In Progress, Review, Done, Blocked を使う
- Spec との対応は Spec 列に記載する
- 依存がある場合は Depends On にチケットIDまたはSpecを記載する
- Story と Task の受け入れ条件は `../story-map.md` の該当セクションを正とする
- Story のタイトルは原則次の形式にする
  - `XXX(想定ユーザー)はXXXできる。なぜならXXXXしたいから。`
- 機能実装タスクは Speckit のフローで管理し、この Backlog では管理しない

## 今後の起票ルール

- 新規チケットは直近番号の次を採番する
- Story と Task は同じ採番系列で管理する
- Epic を追加する場合は、`NNN_エピック名/` のフォルダを作成する
- Phase 1 の全体受け入れ条件と候補チケットは `../story-map.md` を参照する
- すべての Story/Task/Spike は必ず Epic にひも付ける

## 運用ルール

- Story は必ず価値を主語に書く
- Task は技術タスク（リファクタリング、技術検証、基盤整備など）に限定する
- Story と Task は原則 1対1 か 1対N で対応づける
- 週次で優先度と依存を見直す
