# Story Map

## 目的

このドキュメントは、`doc-repo` が提供するユーザー価値を、ユーザー行動を軸に整理するための Story Map である。

機能や技術要素を単に並べるのではなく、利用者がどのような流れでプロダクトを利用し、どの時点で価値を得られるかを明確にする。

## プロダクトゴール

リポジトリ内の Markdown ドキュメントを、Git クライアントやエディタを使わなくても、ブラウザから簡単に閲覧できる状態にする。

## 対象ユーザー

### 生成者

Markdown を含むリポジトリを管理し、閲覧用サイトを生成する人。

主に以下を想定する。

- 開発者
- プロジェクト管理者
- ドキュメント管理者
- 仕様書や議事録を Git 管理しているチームメンバー

### 閲覧者

生成されたサイトを使って、リポジトリ内のドキュメントを読む人。

主に以下を想定する。

- Git の操作に不慣れな人
- Markdown エディタを利用していない人
- 非エンジニアのプロジェクト関係者
- リポジトリのドキュメントをブラウザで確認したい人

## ユーザー体験の流れ

1. 生成準備をする
2. ドキュメントサイトを生成する
3. 生成結果を確認する
4. 閲覧者に共有する
5. 文書の一覧を見る
6. 読みたい文書を探す
7. 文書の内容を読む
8. ドキュメントを更新する

## Walking Skeleton

最初に、以下の一連の流れを最小構成で成立させる。

1. 生成者が対象リポジトリでコマンドを実行する
2. リポジトリ内の Markdown が 1 件読み取られる
3. HTML ファイルが `.doc-repo` に生成される
4. 生成者が HTML をブラウザで開く
5. 閲覧者が文書を選択する
6. 閲覧者が変換された本文を読む

Walking Skeleton では、デザインや設定の充実よりも、CLI から閲覧までの処理が一貫して動作することを優先する。

### Acceptance Criteria

#### STORY-001: ワンコマンドで実行する

- 対象リポジトリのルートで CLI を実行できる
- `npx` 経由でも実行できる
- 必須の設定ファイルを事前に作成しなくても動作する
- コマンドの成否が標準出力から判断できる

#### STORY-002: Markdown を自動収集する

- 実行ディレクトリ配下を再帰的に探索する
- 拡張子が `.md` のファイルを収集する
- ネストしたフォルダ内の Markdown も収集する
- 生成先である `.doc-repo` 配下は収集対象に含めない
- 同じファイルを重複して収集しない

#### STORY-003: Markdown を HTML へ変換する

- Markdown の見出しを HTML へ変換できる
- Markdown の段落を HTML へ変換できる
- Markdown の箇条書きを HTML へ変換できる
- Markdown のコードブロックを HTML へ変換できる
- 変換に失敗した場合、対象ファイルを特定できる

#### STORY-004: 生成物を固定の場所へ出力する

- 生成物はリポジトリ直下の `.doc-repo` へ出力される
- 出力先が存在しない場合は自動で作成される
- 再実行時に古い生成物が不整合な状態で残らない
- 出力完了後に出力先が表示される

#### STORY-005: ブラウザで生成結果を確認する

- 生成されたエントリーポイントとなる HTML を直接開ける
- 閲覧に専用エディタを必要としない
- 閲覧に CLI プロセスの継続起動を必要としない
- サイトの基本 UI が崩れず表示される

#### STORY-006: 生成物を共有する

- `.doc-repo` 配下に閲覧に必要なファイルがまとまっている
- 元の Markdown ファイルを別途渡さなくても閲覧できる
- 閲覧者が npm コマンドを実行しなくても閲覧できる
- 閲覧に Git クライアントを必要としない

#### STORY-007: 文書一覧を見る

- 左ペインにフォルダと Markdown ファイルが表示される
- リポジトリ内のフォルダ階層が維持される
- フォルダとファイルを視覚的に区別できる
- 文書が存在しない場合、その状態が分かる

#### STORY-008: フォルダ構成を辿る

- ネストしたフォルダを階層的に表示できる
- フォルダを開閉できる
- 文書がどのフォルダに属するか判断できる
- 元のリポジトリ構造とナビゲーション構造が対応している

#### STORY-009: 文書を選択する

- ファイル名を選択すると対象文書が表示される
- 現在選択しているファイルが分かる
- 別のファイルを選択すると本文が切り替わる
- 文書の切り替え時にページ全体の再読み込みを必要としない、または閲覧体験を損なわない

#### STORY-010: 本文を読む

- 右ペインに選択した文書の本文が表示される
- 見出し、段落、箇条書き、コードブロックを判別できる
- 長い文書を縦にスクロールして読める
- 左ペインと右ペインを同時に確認できる
- Markdown の元記法ではなく、変換後の表示を基本とする

#### STORY-011: サイトを更新する

- Markdown を変更した後、同じコマンドを再実行できる
- 変更後の内容が生成物に反映される
- 削除された Markdown が生成後の一覧に残らない
- 新しく追加された Markdown が一覧に反映される

## Story と Tech Task の判断基準

### Story として扱うもの

以下を説明できる項目は Story として扱う。

- 誰が利用するか
- 何をしたいか
- それによってどのような価値を得るか
- 完了時にユーザーから見える変化があるか

### Tech Task として扱うもの

以下に該当する項目は Tech Task として扱う。

- 単独ではユーザー価値が成立しない内部実装
- Story を実現するためのモジュール作成
- ビルド、テスト、エラー処理などの技術的な作業
- ライブラリの導入や内部構造の整理
- 将来の変更に備えた責務分離

## 判断原則

- 迷った場合は、ユーザーから見える価値を Story として優先する
- `.md` の探索や HTML 変換そのものは、原則として Tech Task として扱う
- ただし、利用者が得られる結果を表す場合は Story として記述する
- 各リリーススライスは、単独でユーザー操作をデモできる状態にする
- 内部モジュールが完成しただけでは、スライス完了とはしない
- MVP では「読めること」を最優先する
- 将来機能への備えだけを理由に MVP を複雑化しない
- Markdown の収集、変換、サイト生成は、将来再利用できる責務として分離する
- CLI の基本的な入力と出力は、初期段階から安定させる
