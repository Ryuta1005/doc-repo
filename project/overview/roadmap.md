# ロードマップ

## 方針

このプロダクトは、Git と Markdown を正本にしたブラウザベースのドキュメントワークスペースを段階的に作る。

最初に閲覧価値を成立させ、その後にローカルサーバー、自動更新、編集・保存体験へ拡張する。

## Phase 1: MVP(Done)

目的: リポジトリ配下の Markdown をブラウザで読めるようにする

- `.md` ファイルの再帰的な読み取り
- フォルダ構成ベースのツリー生成
- Markdown から HTML への変換
- 単一の静的出力ディレクトリ生成
- 既定の出力先を `.doc-repo` に統一
- 左ツリー / 右本文の基本 UI
- `npx` でも実行できるワンコマンド CLI 体験

## Phase 2: 利便性向上

目的: 日常利用しやすいツールにする

- ローカルサーバー起動
- 変更監視とホットリロード
- 設定ファイル導入（`doc-repo.config.json`）
  - 設定ファイルが見つかればそのディレクトリをルートとして使用
  - 見つからなければ `.git` を上位に向かって探すフォールバック
  - rootDir指定
  - include/exclude 指定
- 設定ファイルの雛形生成コマンド（`doc-repo init`）

## Phase 3: 編集体験

目的: 閲覧ツールから軽量なドキュメントワークスペースへ広げる

- Markdown ショートカット対応のリッチテキスト編集
- 編集内容の Markdown ファイル反映
- ツールバーによる本文、見出し1〜3、太字、イタリックの編集
- チェックボックス状態の永続化（進捗管理）
- 保存後の生成済み表示への自動反映
- 競合や unsafe な更新に対する保護

Backlog 起票方針:

- Epic: 009\_編集保存の最小体験を成立させる
  - 019: React + Hono で Phase 2 までの閲覧・serve・watch 体験を再構成する
  - 010: リッチテキストエディタで文書を編集し、Markdown として保存する
- Epic: 016\_編集保存を安全に運用できるようにする
  - 011: 競合・unsafe 更新の保護
  - 013: 保存失敗理由の可視化
- Epic: 017\_文書操作を便利にする
  - 012: チェックボックス状態の保持
- Epic: 018\_Markdownから関連資料を参照できる
  - 015: Markdown から添付ファイルへリンクする

## 判断原則

- MVP では「読めること」を最優先する
- 将来機能のために、MVP を過度に複雑化しない
- 一方で、Markdown の収集・変換・サイト生成は再利用可能な境界で分離する
- Phase 3 以降は、静的生成そのものではなく `serve` ベースのドキュメントワークスペースを正規入口として扱う
- 破壊的変更を避けるため、CLI の基本入出力は初期から安定させる
