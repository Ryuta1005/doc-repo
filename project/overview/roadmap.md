# ロードマップ

## 方針

このプロダクトは、Git と Markdown を正本にしたブラウザベースのドキュメントワークスペースを段階的に作る。

最初に閲覧価値を成立させ、その後にローカルサーバー、自動更新、編集・保存体験へ拡張する。

Phase 3 までで、`serve` ベースの React + Hono ワークスペース上で Markdown を閲覧・編集・保存する最小体験は完了済みとする。以降は日常利用の操作性を高める拡張フェーズとして扱う。

## Phase 1: MVP(Done)

目的: リポジトリ配下の Markdown をブラウザで読めるようにする

020 で静的生成の直接入口は廃止済み。以下は初期 MVP で成立させた価値と、現在も残す中核処理を示す。

- `.md` ファイルの再帰的な読み取り
- フォルダ構成ベースのツリー生成
- Markdown から HTML への変換
- 左ツリー / 右本文の基本 UI
- ワークスペース起動を正規入口にしたローカル閲覧体験

## Phase 2: 利便性向上(Done)

目的: 日常利用しやすいツールにする

- ローカルサーバー起動
- 変更監視とホットリロード
- 設定ファイル導入（`doc-repo.config.json`）
  - 設定ファイルが見つかればそのディレクトリをルートとして使用
  - 見つからなければ `.git` を上位に向かって探すフォールバック
  - rootDir指定
  - include/exclude 指定
- 設定ファイルの雛形生成コマンド（`doc-repo init`）

## Phase 3: 編集体験(Done)

目的: 閲覧ツールから軽量なドキュメントワークスペースへ広げる

- Markdown ショートカット対応のリッチテキスト編集
- 編集内容の Markdown ファイル反映
- ツールバーによる本文、見出し1〜3、太字、イタリックの編集
- 保存後の Viewer 表示への自動反映

Backlog 起票方針:

- Epic: 009\_編集保存の最小体験を成立させる
  - 019: React + Hono で Phase 2 までの閲覧・serve・watch 体験を再構成する
  - 010: リッチテキストエディタで文書を編集し、Markdown として保存する

完了判断:

- ローカル利用を前提に、保存は「あと勝ち」を許容する
- rootDir 外、`.md` 以外、include / exclude 対象外への保存拒否など、基本ガードは Phase 3 の範囲で満たす

## Phase 4: 文書操作と日常利用の拡張

目的: 既存文書の編集に加え、文書の追加・整理・軽量な状態更新を扱えるようにする

- 新しい Markdown 文書の作成
- 不要な Markdown 文書の削除
- チェックボックス状態の永続化（進捗管理）
- 保存失敗理由の表示整理
- 必要に応じた軽量な外部変更注意

Backlog 起票方針:

- Epic: 021\_文書を作成削除できる
  - 022: 新しい Markdown 文書を作成する
  - 023: 不要な Markdown 文書を削除する
- Epic: 017\_文書操作を便利にする
  - 012: チェックボックス状態の保持
- Epic: 016\_編集保存を安全に運用できるようにする
  - 011: 競合・unsafe 更新の保護
  - 013: 保存失敗理由の可視化
- Epic: 018_Markdownから関連資料を参照できる
  - 015: Markdown から添付ファイルへリンクする

## 判断原則

- MVP では「読めること」を最優先する
- 将来機能のために、MVP を過度に複雑化しない
- 一方で、Markdown の収集・変換・配信は再利用可能な境界で分離する
- 020 以降は、静的生成の直接入口を廃止し、`serve` ベースのドキュメントワークスペースを正規入口として扱う
- 破壊的変更を避けるため、ワークスペース起動コマンドの基本入出力は安定させる
