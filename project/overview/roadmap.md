# ロードマップ

## 方針

このプロダクトは、一気に多機能化するのではなく、閲覧価値を最短で届ける段階的な開発を前提とする。

最初に静的 HTML 生成の価値を成立させ、その後にローカルサーバーや編集体験を拡張する。

## Phase 1: MVP(Done)

目的: リポジトリ配下の Markdown をブラウザで読めるようにする

- `.md` ファイルの再帰的な読み取り
- フォルダ構成ベースのツリー生成
- Markdown から HTML への変換
- 単一の静的出力ディレクトリ生成
- 既定の出力先を `.doc-repo` に統一
- 左ツリー / 右本文の基本 UI
- `npx` でも実行できるワンコマンド CLI 体験

## Phase 2: 利便性向上

目的: 日常利用しやすいツールにする

- ローカルサーバー起動
- 変更監視とホットリロード
- 設定ファイル導入（`doc-repo.config.json`）
  - 設定ファイルが見つかればそのディレクトリをルートとして使用
  - 見つからなければ `.git` を上位に向かって探すフォールバック
  - rootDir指定
  - include/exclude 指定
- 設定ファイルの雛形生成コマンド（`doc-repo init`）

## Phase 3: 編集体験

目的: 閲覧ツールから軽量なドキュメントワークスペースへ広げる

- ブラウザ上での Markdown 編集
- 保存内容のファイル反映
- チェックボックス状態の永続化（進捗管理）
- プレビューと編集の往復改善
- 競合や unsafe な更新に対する保護

Backlog 起票方針（案）:

- Epic: 009\_閲覧ツールを軽量なドキュメントワークスペースへ拡張する
- Story: 010〜013 を Phase 3 のユーザーストーリーとして起票する
  - 010: ブラウザ上での Markdown 編集と元ファイル保存
  - 011: 競合・unsafe 更新の保護
  - 012: チェックボックス状態の保持
  - 013: 保存失敗理由の可視化

## 判断原則

- MVP では「読めること」を最優先する
- 将来機能のために、MVP を過度に複雑化しない
- 一方で、Markdown の収集・変換・サイト生成は再利用可能な境界で分離する
- 破壊的変更を避けるため、CLI の基本入出力は初期から安定させる
