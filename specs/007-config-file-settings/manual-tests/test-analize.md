# 手動テスト分析・設計（Story 007）

## 目的

Story 007 の受け入れ条件と `spec.md` の FR/SC を、人間の操作で end-to-end に検証するための手動テスト設計を定義する。

## 対象範囲

- `doc-repo.config.json` の探索と適用
- `rootDir` の決定（設定ファイル相対 / 省略 / `.git` fallback / `cwd` fallback）
- `include` / `exclude` による収集対象制御
- `generate` と `serve` の設定解決結果一致
- `serve` の監視範囲と生成範囲の一致
- `exclude` 優先規則と既定除外マージ
- 不正設定時のフィールド特定付きエラーメッセージ
- CLI `--port` と設定ファイル `port` の優先順位

## 非対象

- `doc-repo init` による設定ファイル雛形生成
- 新規 CLI オプション（`--root-dir` など）の追加
- ブラウザ編集機能や保存機能
- 認証・外部公開サーバー運用

## テスト観点

1. 設定探索観点

- 上位ディレクトリ探索で最初に見つかった設定ファイルを使うこと
- 設定ファイルなし時のみ `.git` 探索へフォールバックすること

2. rootDir 解決観点

- 相対 `rootDir` を設定ファイル配置ディレクトリ基準で解決すること
- `rootDir` 省略時に設定ファイル配置ディレクトリを使うこと
- 設定なし時に `.git` / `cwd` fallback が働くこと

3. 収集対象一致観点

- `generate` と `serve` で同じ対象集合になること
- watch 対象が生成対象とずれないこと

4. include/exclude ルール観点

- `include` 省略時に `**/*.md` が使われること
- `include: []` が収集対象ゼロを意味すること
- `include: []` が `include` 未指定と同一扱いになること（全 `**/*.md`）
- `exclude` 優先（両方一致は除外）であること
- 利用者 `exclude` が既定除外へ追加されること

5. 失敗ハンドリング観点

- JSON 構文エラー
- `rootDir` 不在/非ディレクトリ
- `include` / `exclude` 型不正
- `port` 範囲外/型不正
- いずれも終了コード 1 かつ対象フィールド・理由表示

6. 互換性観点

- 設定ファイルなしでも従来どおり動作すること
- `--port` が設定ファイル `port` より優先されること

## テストタイプ設計

- E2E 手動テスト: 主要ユーザージャーニー（US1/US2/US3/US4）
- 異常系手動テスト: 不正設定入力の失敗挙動
- 契約確認テスト: `contracts/*.md` と実行結果の整合
- ログ/終了コード確認テスト: CLI 実行結果の観測

## 入口条件

- `npm install` 済み
- `npm run build` 成功済み
- 参照用 Markdown を複数配置した検証用ディレクトリを用意
- 既存の `doc-repo serve` が停止済み
- ターミナルで終了コード確認が可能

## 終了条件

- 全手動ケースを実行済み
- Fail が 0 件、または Fail について再現手順と原因を記録済み
- FR-001〜FR-018 と SC-001〜SC-005 の全項目に最低 1 件のケース紐付けあり

## リスクと補完策

- リスク: ローカルファイル構成差で fallback 判定がぶれる
- 補完策: 各ケースで作業ディレクトリ構成を明示し、実行前に `pwd` と配置を確認する

- リスク: include/exclude の glob 解釈差を見誤る
- 補完策: 期待対象ファイル一覧をケースごとに先に定義してから実行する

- リスク: 失敗ケースで複数エラーが同時発生し原因が曖昧になる
- 補完策: 1 ケース 1 不正値に限定して検証する

## 実施ログの記録方針

各ケースで以下を記録する。

- 実施日時
- 実施者
- 実行コマンド
- 標準出力/標準エラー要点
- 終了コード
- 判定（Pass/Fail/Blocked）
- 補足（再現条件、改善メモ）
