# Feature Specification: Markdown文書削除

**Feature Branch**: `023-delete-markdown-doc`  
**Created**: 2026-06-21  
**Status**: Draft  
**Input**: User description: "project/planning/backlog/021*文書を作成削除できる/023*利用者は不要なMarkdown文書を削除できる.md"

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 不要な文書を削除して一覧を整理する (Priority: P1)

利用者はサイドバー上でファイル名またはフォルダ名にマウスホバーしたときに表示されるミートボールメニューから、不要な Markdown 文書またはフォルダを削除し、一覧を整理できる。

**Why this priority**: 既存文書を整理できないと、ブラウザ上で文書ワークスペースを完結して運用する価値が不足するため。

**Independent Test**: 利用者がサイドバーのファイル行またはフォルダ行にホバーしてミートボールメニューを開き、ゴミ箱アイコン付きの `削除` を選び、確認ダイアログで対象名と警告文を確認したうえで削除を完了し、文書ツリー更新と残存項目への自然な遷移まで確認できれば、この価値を単独で検証できる。

**Acceptance Scenarios**:

1. **Given** 利用者がサイドバー上のファイル名またはフォルダ名にマウスホバーしている, **When** 行末を確認する, **Then** ミートボールメニューが表示される。
2. **Given** 利用者がミートボールメニューを開いている, **When** メニュー内容を確認する, **Then** ゴミ箱アイコンと `削除` 文言を持つ削除項目が表示される。
3. **Given** 利用者が削除項目をクリックした, **When** 確認ダイアログが表示される, **Then** h4 相当の見出し `削除しますか？`、対象名を識別できる表示、および `Git Commitなどをしていない場合、元に戻せません。` が表示される。
4. **Given** 削除確認が表示されている, **When** 利用者が `削除` を選ぶ, **Then** 赤背景白文字の削除ボタンで削除を確定できる。
5. **Given** 削除確認が表示されている, **When** 利用者が `キャンセル` を選ぶ, **Then** 現在の未保存変更確認ダイアログと同じデザインのキャンセルボタンで削除を中止できる。
6. **Given** 利用者がファイルを対象に削除を確定する, **When** 削除処理が成功する, **Then** 対象の Markdown 文書が削除される。
7. **Given** 利用者がフォルダを対象に削除を確定する, **When** 削除処理が成功する, **Then** そのフォルダ自身と配下の管理対象 Markdown 文書群がまとめて削除される。
8. **Given** 削除が成功した, **When** ワークスペースが更新される, **Then** 文書ツリーから対象項目が消え、残っている項目へ自然に遷移する。

### User Story 2 - 不正または不適切な削除を安全に拒否する (Priority: P2)

利用者は削除できない対象に対して誤って操作しても、rootDir 外や管理対象外のファイルを消さず、失敗理由を確認できる。フォルダ削除時は、配下に管理対象外ファイルまたは削除不許可の項目が 1 つでも含まれていれば、フォルダ全体の削除が拒否される。

**Why this priority**: 削除は破壊的操作であり、対象外ファイルや不正パスを確実に守ることが運用上の安全性に直結するため。

**Independent Test**: rootDir 外、`.md` 以外、include 対象外、exclude 対象、存在しない文書への削除要求に加え、配下に管理対象外ファイルを含むフォルダ削除要求を個別に実施し、すべて拒否され理由が表示されることを確認できれば独立検証できる。

**Acceptance Scenarios**:

1. **Given** 削除要求の対象が rootDir 外を指している, **When** 削除を実行する, **Then** 削除は拒否され理由が表示される。
2. **Given** 削除要求の対象が `.md` 以外のファイルである, **When** 削除を実行する, **Then** 削除は拒否され理由が表示される。
3. **Given** 削除要求の対象が include 条件を満たさない、または exclude 条件に該当する, **When** 削除を実行する, **Then** 削除は拒否され理由が表示される。
4. **Given** 削除要求に `..` 等を含む不正なパス移動がある, **When** 削除を実行する, **Then** 削除は拒否される。
5. **Given** 削除要求の対象ファイルがすでに存在しない, **When** 削除を実行する, **Then** 削除は失敗し存在しないことを示す理由が表示される。
6. **Given** 利用者がフォルダ削除を実行しようとしている, **When** その配下に管理対象外ファイル、`.md` 以外のファイル、include 対象外または exclude 対象の項目が 1 つでも含まれる, **Then** フォルダ全体の削除は拒否され理由が表示される。

---

### User Story 3 - 未保存変更との整合を保って削除する (Priority: P3)

利用者は編集中の未保存変更がある状態でも、既存の未保存変更確認を尊重したうえで削除操作を安全に進められる。

**Why this priority**: 既存の編集保護を壊すと削除機能の追加によって変更消失リスクが増えるため。

**Independent Test**: 未保存変更がある文書または画面状態から削除操作を開始し、既存の未保存変更確認フローが先に適用され、その結果に応じて削除継続または中止されることを確認できれば独立検証できる。

**Acceptance Scenarios**:

1. **Given** 利用者に未保存変更がある, **When** 削除操作を開始する, **Then** 既存の未保存変更確認フローが尊重される。
2. **Given** 未保存変更確認で削除継続が許可された, **When** 利用者が削除確認で削除を確定する, **Then** 対象文書の削除が実行される。
3. **Given** 未保存変更確認で操作中止を選んだ, **When** 削除処理へ進もうとする, **Then** 削除は実行されない。

### Edge Cases

- 最後の 1 件である文書を削除した場合でも、空状態が分かる形でワークスペースが更新される。
- 削除確認を表示した後に対象ファイルが外部要因で消えていた場合、削除は失敗として扱われ理由が表示される。
- 同名の文書が別フォルダに存在する場合でも、確認内容から削除対象を取り違えない。
- フォルダ削除対象の直下または子孫に、画像・PDF・設定ファイルなど管理対象外ファイルが 1 つでも含まれる場合は、フォルダ全体を削除せず拒否する。
- 空フォルダを削除した場合は、そのフォルダ自体が削除される。
- フォルダ削除に成功した場合は、配下に残骸となる空ディレクトリを残さない。
- include と exclude が競合するパスでは、削除可否ルールが一貫して適用される。
- 削除対象が現在表示中の文書である場合でも、削除後に存在しない文書表示へ留まらない。
- 削除確認を開いたまま別操作を試みた場合でも、確認中の対象が意図せず別文書へ切り替わらない。
- フォルダ削除確認では、対象フォルダ名を識別でき、配下がまとめて削除対象になることが分かる。

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: システムは、サイドバー上のファイル名またはフォルダ名にマウスホバーしたときだけ、その行の末尾にミートボールメニューを表示しなければならない。
- **FR-002**: システムは、ミートボールメニューを開いたとき、ゴミ箱アイコンと `削除` 文言を持つ削除項目を表示しなければならない。
- **FR-003**: システムは、ファイルとフォルダの両方に対して削除操作の入口を提供しなければならない。
- **FR-004**: システムは、削除操作の実行前に確認ダイアログを表示しなければならない。
- **FR-005**: システムは、削除確認内で対象ファイル名または対象フォルダ名を利用者が識別できる形で表示しなければならない。
- **FR-006**: システムは、削除確認内に h4 相当の見出し `削除しますか？`、対象名、および `Git Commitなどをしていない場合、元に戻せません。` を表示し、本文として `本当に削除しますか？` を重複表示してはならない。
- **FR-007**: システムは、削除確認内で赤背景白文字の `削除` ボタンを表示しなければならない。
- **FR-008**: システムは、削除確認内で非破壊的な操作を示す（`.btn-ghost` 相当の視覚スタイル）`キャンセル` ボタンを表示しなければならない。
- **FR-009**: 利用者は、削除確認で削除を確定または中止できなければならない。
- **FR-010**: システムは、ファイル削除対象を rootDir 配下の `.md` ファイルに限定しなければならない。
- **FR-011**: システムは、フォルダ削除対象を rootDir 配下のフォルダに限定しなければならない。
- **FR-012**: システムは、`..` 等によるパストラバーサルを含む削除要求を拒否しなければならない。
- **FR-013**: システムは、include 条件を満たし、かつ exclude 条件に該当しない Markdown ファイルに対してのみ削除を許可しなければならない。
- **FR-014**: システムは、フォルダ削除時に配下の管理対象 Markdown を再帰的な削除対象として扱わなければならない。
- **FR-015**: システムは、フォルダ削除時に配下へ管理対象外ファイル、`.md` 以外のファイル、include 対象外または exclude 対象の項目が 1 つでも含まれる場合、フォルダ全体の削除を拒否しなければならない。
- **FR-016**: システムは、対象ファイルまたは対象フォルダが存在しない場合に削除を失敗として扱い、失敗理由を表示しなければならない。
- **FR-017**: システムは、削除成功後に文書ツリーを更新し、削除済みファイルまたは削除済みフォルダを一覧から除去しなければならない。
- **FR-018**: システムは、削除成功後に残っている項目へ自然に遷移するか、残存項目がない場合は空状態を表示しなければならない。
- **FR-019**: システムは、未保存変更がある状態で削除操作が開始された場合、既存の未保存変更確認フローを削除確認より先に適用しなければならない。
- **FR-020**: システムは、未保存変更確認で操作中止が選択された場合、削除処理へ進んではならない。
- **FR-021**: システムは、削除対象が現在表示中の文書またはその文書を含むフォルダである場合でも、削除後に存在しない項目を選択状態として保持してはならない。（FR-018 の補足: 残存文書がない場合の空状態表示によって同条件を満たせる）
- **FR-022**: システムは、削除に失敗した場合、利用者が再試行または状態確認の判断に使える失敗理由を表示しなければならない。
- **FR-023**: システムは、フォルダ削除に成功した場合、削除対象フォルダ自体も除去し、配下に空ディレクトリを残してはならない。
- **FR-024**: システムは、本 Story でゴミ箱移動、論理削除、復元機能を提供してはならない。

### Key Entities _(include if feature involves data)_

- **DeletionTarget**: 利用者が削除対象として選ぶ項目。主な属性は targetType（file/folder）、displayName、resolvedPath。
- **DocumentDeletionRequest**: 利用者が削除操作で確定した対象情報。主な属性は target、requestedAt、requestSource。
- **DeletionConfirmation**: 利用者へ削除前に提示される確認状態。主な属性は displayName、targetType、warningMessage、confirmable、cancelable。
- **DeletionPolicy**: 削除可否を判断するルール集合。主な属性は rootDir 制約、許可拡張子制約、include 条件、exclude 条件、存在確認、フォルダ配下検査結果。
- **DocumentDeletionResult**: 削除処理の結果。主な属性は status、reasonCode、affectedTargets、nextSelectionState。
- **WorkspaceSelectionState**: 削除後の画面遷移先を表す状態。主な属性は selectedItem、fallbackItem、emptyState。

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 代表的な利用者シナリオにおいて、利用者の 90% 以上が 30 秒以内に削除対象の確認から削除完了まで到達できる。
- **SC-002**: rootDir 外、`.md` 以外、include 対象外、exclude 対象、パストラバーサルを含む不正削除要求は 100% 拒否される。
- **SC-003**: 存在しない文書への削除要求で、100% の試行に失敗理由が表示される。
- **SC-004**: 削除成功ケースで、100% の試行において文書ツリー更新と削除済み文書の非表示化が 2 秒以内に完了する。
- **SC-005**: 削除成功後、100% の試行で残存文書への遷移または空状態表示のどちらかが成立し、存在しない文書が選択されたまま残らない。
- **SC-006**: 未保存変更ありの削除開始ケースで、100% の試行において既存の未保存変更確認が先に適用される。
- **SC-007**: フォルダ削除要求のうち、配下に管理対象外ファイルまたは削除不許可項目を含むケースは 100% 拒否される。
- **SC-008**: 一時的 I/O 失敗（TRANSIENT_IO）による削除失敗ケースで、100% の試行に失敗理由と再試行可能であることが表示される。

## Assumptions

- 対象利用者は rootDir 配下の管理対象 Markdown 文書に対する削除権限を持つ。
- 本 Story の対象は単一 Markdown 文書削除と単一フォルダ削除であり、複数選択一括削除は対象外とする。
- 削除操作はサイドバー上のホバー時ミートボールメニューから開始する。
- 削除後の「自然な遷移」は、残存文書がある場合は利用者が作業を継続しやすい既存文書へ遷移し、残存文書がない場合は空状態を表示することを指す。
- 削除対象の確認はファイル名を必須表示とし、必要に応じて取り違え防止に役立つ追加情報を併記してよい。
- フォルダ削除は、配下に管理対象 Markdown 以外を含まない場合に限り、そのフォルダ自体と配下項目をまとめて削除する。
- ゴミ箱や復元機能は初期段階では扱わず、必要な復旧は Git で行う前提とする。
