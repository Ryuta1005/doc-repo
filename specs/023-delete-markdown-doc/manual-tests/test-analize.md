# 手動テスト分析・設計（Story 023）

## 目的

Story 023 の受け入れ条件と `spec.md` の FR/SC を、人間の操作で end-to-end に確認できるようにする。対象は、サイドバーのホバー時ミートボールメニューから開始する Markdown 文書削除とフォルダ削除、およびそれに伴う安全性・表示整合・選択状態更新である。

## 対象範囲

- サイドバー上でファイル行・フォルダ行にホバーしたときの削除導線表示
- ゴミ箱アイコン付き `削除` 項目の表示
- 削除確認ダイアログの文言、対象名表示、確定/取消ボタン
- ファイル削除の成功フロー
- フォルダ削除の成功フロー
- rootDir 外、`.md` 以外、include/exclude 条件外、path traversal、存在しない対象の拒否
- フォルダ配下に管理対象外ファイルまたは削除不許可項目が混在する場合の拒否
- 未保存変更確認フローとの優先関係
- 削除成功後のツリー更新、自然な遷移、空状態表示

## 非対象

- ゴミ箱移動、論理削除、復元機能
- 複数選択の一括削除
- include/exclude 設定そのものの編集
- 編集画面での Markdown 保存機能の詳細検証
- API レベルの自動テスト

## テスト観点

1. 導線表示観点

- ホバー時のみミートボールメニューが出ること
- ファイル行とフォルダ行の両方で削除導線があること

2. 確認ダイアログ観点

- 対象名が識別可能に出ること
- 警告文が仕様どおり出ること
- 削除とキャンセルが視覚的に区別できること

3. 成功フロー観点

- ファイル削除で対象が消えること
- フォルダ削除で配下の管理対象 Markdown とフォルダ自体が消えること
- 成功後に一覧と現在選択が整合すること

4. 安全性観点

- rootDir 外や不正パスが拒否されること
- `.md` 以外や include/exclude 条件外が拒否されること
- 存在しない対象が失敗理由付きで拒否されること
- フォルダ配下に管理対象外ファイルがある場合、全体削除が拒否されること

5. 状態整合観点

- 未保存変更があるとき、削除確認より先に既存ガードが出ること
- 削除後に存在しない文書を選択状態として残さないこと
- 最後の 1 件削除時は空状態が成立すること

## テストタイプ設計

- E2E 手動テスト: 主要な削除ジャーニーの確認
- 異常系手動テスト: 対象不正、存在しない対象、フォルダ混在拒否
- 視覚確認テスト: メニュー、確認ダイアログ、ボタンデザイン
- 状態遷移確認テスト: ツリー更新、選択遷移、空状態

## 入口条件

- `npm install` 済み
- `npm run build` 成功済み
- 対象ワークスペースに削除用の `.md` とフォルダが用意されている
- フォルダ削除の失敗ケース用に、管理対象外ファイルを含むフォルダを用意できる
- 既存の `doc-repo serve` が停止済み
- ブラウザで表示 URL へアクセスできる

## 終了条件

- 全手動ケースを実行済み
- 各ケースで実行結果が記録済み
- Fail または Blocked が発生した場合、再現条件と観測内容が残っている
- FR-001 から FR-023 までのうち、対象範囲に含めた要件に最低 1 件のケースが紐付いている
- SC-001 から SC-007 までのうち、対象範囲に含めた要件に最低 1 件のケースが紐付いている

## リスクと補完策

- リスク: 削除対象の用意が不十分だと成功/失敗の両方を確認しづらい
- 補完策: 成功用の純粋な Markdown フォルダと、失敗用の混在フォルダを事前に用意する

- リスク: 末尾の選択遷移は文書数によって見え方が変わる
- 補完策: 1 件のみのケースと複数件残存ケースを分けて確認する

- リスク: ブラウザや OS によりホバー表示の感覚が異なる
- 補完策: メニュー表示の有無を視覚で確認し、時間測定は応答完了時刻で記録する

- リスク: フォルダ削除の拒否理由は配下構成によって変わる
- 補完策: どの項目が拒否原因かを手順ごとに記録する

## 実施ログの記録方針

各ケースで以下を記録する。

- 実施日時
- 実施者
- 実行コマンド
- 対象パス
- 標準出力または画面上の要点
- 判定
- 補足

判定と結果の記録は人間が実施後に埋める前提とし、事前入力しない。
