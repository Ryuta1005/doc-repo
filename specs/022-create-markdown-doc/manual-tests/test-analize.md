# 手動テスト分析・設計（Story 022）

## 目的

Story 022「利用者は新しい Markdown 文書を作成できる」について、利用者が実際の UI 操作と最小限のファイル確認で受け入れ条件を検証できる手動テスト方針を定義する。特にホバー `+` 導線、作成先解決、入力制約、安全な拒否、未保存変更ガード維持を重点確認する。

## 対象範囲

- サイドバーのファイル/フォルダ行ホバー時 `+` 表示
- `+` クリックから新規作成画面遷移
- 作成先解決（フォルダ起点/ファイル起点）
- ファイル名入力ルール（ファイル名本文のみ、末尾 `.md` 常時付与）
- 不正入力拒否（`/` `\\`、path traversal、rootDir 外、include/exclude 対象外）
- 重複時の上書き防止
- 作成成功時のツリー更新と自動選択
- 内部ファイル名 `.md` 保存とサイドバーの末尾 `.md` なし表示（既存文書 + 新規文書）
- 未保存変更ガードと作成開始フローの両立

## 非対象

- 文書削除機能（Story 023）
- リッチテキスト編集品質そのもの
- `serve/watch` の性能最適化全般
- ブラウザデザインの主観評価

## テスト観点

1. 導線・操作性観点

- ホバー時のみ `+` が表示される
- クリックで作成画面へ遷移する

2. 作成先解決観点

- フォルダ起点は当該フォルダ直下
- ファイル起点は親フォルダ直下（同階層）

3. 入力ルール観点

- 入力値の末尾に常に `.md` を補完
- `.md` 明示時もファイル名本文として扱い、保存用 `.md` を付与
- 区切り文字含み入力を拒否

4. セキュリティ・安全性観点

- rootDir 外・`..` を含む解決を拒否
- include/exclude 条件外を拒否
- 既存同名への上書きを禁止

5. 表示整合観点

- 内部保存名は `.md`
- サイドバー表示名は既存文書/新規文書とも最後の `.md` なし

6. 応答時間観点

- 作成成功時の `t1 - t0` を 10 回計測し、p95 が 2 秒以内である

7. 既存保護観点

- 未保存変更確認フローが維持される

## テストタイプ設計

- E2E 手動テスト: UI 上の操作と表示結果を観測
- 契約整合確認: `contracts/http-document-create-contract.md` と `contracts/viewer-sidebar-create-contract.md` の期待挙動と照合
- ファイルシステム確認: 作成結果ファイル存在、上書き非発生を確認

## 入口条件

- リポジトリルートで依存関係インストール済み
- ビルド済み
- `node dist/cli/index.js serve` で編集ワークスペースを起動できる
- テスト用 rootDir に読み書き権限がある
- include/exclude の検証用に対象内/対象外パスを事前準備できる

推奨コマンド:

```bash
npm install
npm run build
node dist/cli/index.js serve
```

## 終了条件

- MT-2201〜MT-2209 を全実行済み
- FR-001〜FR-021 が少なくとも 1 ケース以上でカバーされる
- SC-001〜SC-007 が少なくとも 1 ケース以上でカバーされる
- Fail/Blocked がある場合、再現条件と暫定対処を記録済み

## リスクと緩和策

- リスク: include/exclude 条件の再現が環境依存でぶれる
- 緩和策: 実施前に使用設定ファイルを記録し、同一設定で全ケースを実行する

- リスク: ツリー更新 2 秒以内（SC-005）の体感判定が主観に寄る
- 緩和策: `t1 - t0` を 10 回記録し p95 で判定する。必要に応じて画面録画で補助する

- リスク: 未保存ガードの分岐が他機能の状態に影響される
- 緩和策: ケース開始前に「未保存あり/なし」状態を明示的に作ってから実施する
