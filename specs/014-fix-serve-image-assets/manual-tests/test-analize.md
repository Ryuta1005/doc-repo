# 手動テスト分析・設計（Story 014）

## 目的

Story 014「serve時の相対画像表示不具合修正」について、利用者が実際のターミナル操作とブラウザまたはHTTP確認で検証できる観点に絞って、受け入れ条件・機能要件（FR）・成功基準（SC）を検証する手動テスト方針を定義する。

## 対象範囲

- Markdown 内の相対画像参照が `.doc-repo` 配下の生成資産を指すこと
- 参照元画像ファイルが `.doc-repo` 配下へコピーされること
- `doc-repo serve` 経由で画像ファイルを HTTP 200 として取得できること
- 同じ生成HTMLを `file://` で開いても画像が表示されること
- 外部URL画像・ハッシュ参照・Markdownページリンクの既存挙動が維持されること

## 非対象

- Markdown 変換結果のHTML構造全般
- CSS 内の `url(...)` 参照の収集
- 通常リンクで参照される PDF / CSV / ZIP などの添付ファイル配信
- HTML 生タグ内の独自属性の解析
- 画像そのものの画質・内容確認
- ブラウザUIの詳細な見た目や操作性

## テスト観点

1. 生成画像観点

- Markdown の相対画像参照が `.doc-repo` 配下のURLへ変換される
- 参照された画像ファイルが `.doc-repo/assets/` 配下にコピーされる

2. serve配信観点

- `doc-repo serve` で生成サイトを開いたとき、相対画像が表示される
- 画像URLを直接HTTP取得した場合に `200` が返る
- `.doc-repo` 外のファイルを直接配信しない

3. file://互換観点

- `.doc-repo` のHTMLを直接開いても、同じ相対画像が表示される

4. 既存挙動維持観点

- 外部URL画像はコピーされず、そのままHTMLへ出力される
- 通常リンクで参照される非Markdownファイルは本チケットではコピー対象にしない
- 実在 Markdown ページへのリンクは従来どおり `.html` リンクへ変換される
- テンプレート資産（`styles.css`、`app.js`）とページ遷移が壊れない

## テストタイプ設計

- E2E手動テスト: 利用者の主要操作シナリオ（generate/serve/file://確認）
- 配信確認テスト: 画像URLのHTTPステータス確認
- 回帰観点テスト: 外部URL、ページリンク、テンプレート資産の既存挙動確認

## 要件トレーサビリティ

| 手動テストID | 主観点                             | 対応FR                 | 対応SC          |
| ------------ | ---------------------------------- | ---------------------- | --------------- |
| MT-1401      | 相対画像の生成画像コピー確認       | FR-001, FR-002         | SC-001, SC-002  |

## 入口条件

- 依存関係インストール済み（`npm install`）
- ビルド済み（`npm run build`）
- `node dist/cli/index.js` で CLI を実行できる
- `curl` またはブラウザの開発者ツールでHTTPステータスを確認できる
- 検証用ディレクトリと画像ファイルを作成できる
- 既存の `doc-repo serve` プロセスが停止している

## 終了条件

- MT-1401〜MT-1406 の全ケースを実行済み
- すべての FR-001〜FR-006 に最低1件の対応ケースが存在
- すべての SC-001〜SC-004 に最低1件の対応ケースが存在
- Fail/Blocked がある場合、再現条件と対処方針が記録されている

## リスクと緩和策

- リスク: ブラウザキャッシュにより修正前後の画像表示結果が混在する
- 緩和策: 検証用ディレクトリを毎回新規作成し、必要に応じてキャッシュ無効化または画像URLの直接HTTP確認を行う

- リスク: `serve` のポート競合で手動確認が開始できない
- 緩和策: `--port` で未使用ポートを指定し、起動URLをログから確認する

- リスク: `file://` の目視確認だけでは画像取得元が曖昧になる
- 緩和策: 生成HTMLの `src` と `.doc-repo/assets/` 配下の実ファイル存在を併せて確認する

## ログ記録方針

各ケースで以下を記録する。

- 実施日時
- 実施者
- 実行コマンド
- 標準出力要点
- 標準エラー要点
- HTTPステータス
- 終了コード
- 判定
- 補足（再現条件、原因、次アクション）
