# 手動テスト分析・設計（Story 019）

## 目的

Story 019「React + Hono ワークスペース基盤」について、既存 Phase 2 の利用者体験を維持できているかを人手で確認するための観点・実施条件・判定基準を定義する。

## 対象範囲

- `doc-repo serve` 起動と閲覧体験（左ツリー、本文表示、文書切替）
- HTTP API 契約（`GET /api/documents`, `GET /api/document?path=...`）
- `path` の入力検証（空文字、`..`、`rootDir` 外参照）と日本語/空白パスの取得成功
- React Viewer の URL ルーティング（URL直打ち、ブラウザリロード、`.md` / `.html` URL、ディレクトリURL）
- React UI、HTTP API、SSE、静的アセットの同一 port / 同一 origin 配信
- 画像・静的アセット配信
- watch による変更検知と SSE `reload` 自動更新
- エラー表示と継続利用性（400/404/500, SSE 切断）
- 静的生成（`doc-repo [scopePath]`）とオフライン閲覧
- 性能測定記録の必須項目と 20% 判定ゲートの記録運用

## 非対象

- 保存 API 本体の実装確認（Phase 3 対象）
- WYSIWYG 編集機能
- 高度な検索・タグ管理
- 019 で定義されていない新規 UX 改善施策

## テスト観点

1. 機能同等性観点（SC-001, SC-004, SC-007）

- 既存の主要導線が移行後も同期待結果で成立する
- 静的生成の主要契約とオフライン閲覧方法を維持する

2. 契約準拠観点（FR-007, FR-008, FR-012）

- 文書一覧/取得 API の契約と識別子仕様が一致する
- 不正入力のエラー契約が守られる
- 日本語または空白を含む相対パスが query parameter 方式で取得できる

3. 監視・通知観点（FR-005, SC-004）

- 変更・追加・削除時に再生成成功でのみ `reload` が通知される
- 失敗時に不要な自動更新を行わない

4. 安定性観点（FR-006, FR-013, SC-004）

- 404/切断/削除中操作でクラッシュせず回復可能
- URL直打ち、リロード、ディレクトリリンクで 500 を出さない
- `serve` がReact Viewerを配信しつつ、静的生成物を破壊的に上書きしない

5. 性能記録観点（SC-002, SC-003, SC-005）

- baseline/new それぞれウォームアップ1回 + 本計測5回を記録する
- 中央値比較、20%以上悪化時の追加記録と承認有無を残す

## テストタイプ設計

- E2E 手動テスト: 起動から閲覧、監視、静的生成、オフライン閲覧までの導線
- 契約確認テスト: API 応答形、エラーコード、SSE payload 形式
- 回帰比較テスト: 既存体験と移行後体験の同期待結果確認
- 測定記録監査テスト: 性能記録テンプレートの充足確認

## 入口条件

- `npm install` 済み
- `npm run build` 済み
- `node dist/cli/index.js --help` が成功する
- 検証用 fixture（通常規模・大量規模・日本語または空白を含むファイル名）が準備済み
- baseline 対象コミットが特定済み
- 計測対象マシン/環境情報を記録できる

## 終了条件

- MT-1901 から MT-1918 を実行済み
- FR-001 から FR-013 が少なくとも 1 ケースに紐づく
- SC-001 から SC-007 が少なくとも 1 ケースに紐づく
- T046-T050 が完了し、SC-002/SC-003/SC-005 の測定記録で必須項目が空欄でない
- 20%以上悪化項目がある場合、追加記録と承認状態が記録されている

## リスクと緩和策

- リスク: watch 通知は環境負荷で遅延し、誤判定しやすい
- 緩和策: 同一マシン・同一 fixture・同一手順で baseline/new を実施し、中央値で比較する

- リスク: ネットワーク切断試験の再現方法が端末依存
- 緩和策: 開発者ツールの offline モードまたはサーバー停止を手順に明記し、再現方法を実測欄へ記録する

- リスク: API 契約と UI 挙動の責務混在で原因切り分けが難しい
- 緩和策: API 単体確認ケースと UI 導線ケースを分離して記録する

- リスク: React Viewer bundle ではなく旧静的アセットで確認してしまう
- 緩和策: `serve` が同一 port から返す `/app.js` と `/styles.css` に React Viewer 用の `viewer-layout` が含まれることを MT-1901 で確認する

- リスク: SPA的なURL遷移は動くが、ブラウザリロードやURL直打ちで静的配信層が404/500を返す
- 緩和策: Markdown URL直打ち/リロード、`.html` URL、ディレクトリURLを MT-1915〜MT-1917 で個別確認する

- リスク: `serve` 起動時に `.doc-repo` の静的生成物をReact成果物で上書きし、静的生成互換確認とReact配信確認が混ざる
- 緩和策: HTTP配信内容と `.doc-repo` 上の実ファイルを MT-1918 で分離して確認する

## ログ記録方針

各ケースで次を記録する。

- 実施日時
- 実施者
- 実行コマンド
- 期待結果との差分
- 標準出力/標準エラー要点
- HTTP ステータス（該当時）
- 判定
- 補足（再現条件、暫定回避、次アクション）

性能計測ケース（MT-1912）では次を必須記録とする。

- `baseline commit`
- `new implementation commit`
- `machine/environment`
- `fixture`
- `measurement target`
- `warmup result`
- `individual results`
- `median`
- `difference`
- `decision`

20%以上悪化項目では次を追加必須とする。

- `user impact`
- `reason not fixed in 019`
- `approval`
