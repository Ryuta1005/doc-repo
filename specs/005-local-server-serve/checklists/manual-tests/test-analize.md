# 手動テスト分析・設計（Story 005）

## 目的

Story 005 の受け入れ条件と `spec.md` の FR/SC を、人間の目でエンドツーエンドに検証するための手動テスト設計を定義する。

## 対象範囲

- `doc-repo serve` の実行フロー
- 初回生成 -> サーバー起動 -> 監視開始の順序
- ポート解決優先順位（CLI > 設定ファイル > デフォルト）
- 失敗時のメッセージと終了コード
- ブラウザでの表示確認

## 非対象

- 監視詳細（再生成トリガーやホットリロード実装詳細）
- 外部公開運用（TLS、認証、アクセス制御）

## テスト観点

1. 正常起動観点

- 生成物が未作成でも `serve` 1 回で閲覧可能になること
- URL が標準出力に表示されること
- ブラウザでサイトが表示されること

2. 設定解決観点

- `port` 未指定時に `4000` を使うこと
- 設定ファイル指定時にその値を使うこと
- CLI 指定が設定ファイルを上書きすること

3. 異常系観点

- 初回生成失敗時にサーバーを起動しないこと
- ポート競合時に理由付きで失敗すること
- 不正な `port` 入力を起動前に弾くこと

4. 終了制御観点

- Ctrl+C で停止できること
- 停止後に同一ポートで再起動できること

5. 可観測性観点

- 成功/失敗の状態遷移が標準出力・標準エラーで判断できること

## テストタイプ設計

- E2E 手動テスト: 主要ユーザージャーニーを通しで確認
- 異常系手動テスト: 失敗要因ごとに独立確認
- 視覚確認テスト: ブラウザ表示の成立確認

## 入口条件

- `npm install` 済み
- `npm run build` 成功済み
- テスト用リポジトリ配下に最低1つ以上の `.md` が存在
- 既存の `doc-repo serve` プロセスが停止済み

## 終了条件

- すべての手動テストケースが実行済み
- Fail が 0 件、または Fail について原因と再現手順が記録済み
- FR と SC の全項目に少なくとも1件以上のテスト紐付けがある

## リスクと補完策

- リスク: 手動実行者による操作差で結果が揺れる
- 補完策: コマンドと期待結果を具体的に固定し、結果記録欄を必須にする

- リスク: ポート競合の再現が環境依存
- 補完策: 明示的に占有プロセスを起動する手順を固定する

- リスク: 監視機能の仕様混在
- 補完策: Story 005 では「監視開始メッセージ確認」のみ対象とし、再生成動作は Story 006 側で検証する

## 実施ログの記録方針

各ケースで以下を記録する。

- 実行日時
- 実行者
- 実行コマンド
- 実際の標準出力/標準エラー（要点）
- 判定（Pass/Fail）
- Fail 時の再現性と補足
