# 手動テスト分析・設計（Story 008）

## 目的

Story 008「設定ファイル雛形の生成（doc-repo init）」について、利用者が実際のターミナル操作で確認できる観点に絞って、受け入れ条件・機能要件（FR）・成功基準（SC）を検証する手動テスト方針を定義する。

## 対象範囲

- `doc-repo init` の正常系（未作成環境での雛形生成）
- 生成される `doc-repo.config.json` のキーとデフォルト値
- 既存ファイル存在時の上書き防止
- 失敗時メッセージと終了コード
- 生成設定を使った `doc-repo` の成立確認

## 非対象

- `doc-repo` の詳細変換品質（HTML構造の細部）
- `doc-repo serve` の配信・watch 挙動
- 将来機能（`--force` 等）
- ブラウザUIの見た目や操作性

## テスト観点

1. 生成観点

- `init` 実行で設定ファイルが新規作成される
- 作成パスが標準出力に表示される

2. 内容妥当性観点

- `rootDir`、`include`、`exclude`、`port` が正しいデフォルトで出力される

3. 保護観点

- 既存 `doc-repo.config.json` が上書きされない
- 既存時メッセージが表示され、終了コード `0` で終わる

4. 失敗ハンドリング観点

- 書き込み不可時に理由付きエラーが表示される
- 終了コード `1` で終わる

5. 実運用成立観点

- 生成した設定ファイルで `doc-repo` が完了する

## テストタイプ設計

- E2E手動テスト: 利用者の主要操作シナリオ（生成、再実行、失敗）
- 契約確認テスト: `contracts/cli-init.md` の出力契約・終了コード契約確認
- 回帰観点テスト: 生成済み設定を使った通常生成の実行成立

## 入口条件

- 依存関係インストール済み（`npm install`）
- ビルド済み（`npm run build`）
- `node dist/cli/index.js` で CLI を実行できる
- 作業端末で権限変更（`chmod`）が可能
- 検証用の空ディレクトリと Markdown 1件以上を準備済み

## 終了条件

- MT-801〜MT-805 の全ケースを実行済み
- すべての FR-001〜FR-007 に最低1件の対応ケースが存在
- すべての SC-001〜SC-004 に最低1件の対応ケースが存在
- Fail/Blocked がある場合、再現条件と対処方針が記録されている

## リスクと緩和策

- リスク: ローカル権限設定の差異で失敗ケースが再現しにくい
- 緩和策: 書き込み不可ケースは一時ディレクトリを作り、`chmod` で明示的に権限を落として実行する

- リスク: 既存ファイル比較で差分確認が曖昧になる
- 緩和策: 再実行前にファイル内容ハッシュを記録し、再実行後に同値を確認する
