# 手動テスト分析: Story 010 リッチテキスト編集と Markdown 保存

## 目的

Story 010 の受け入れ条件と FR/SC を、人間の操作で観測できるエンドツーエンド観点で検証する。
特に「編集開始から保存完了までの最小体験」「未保存変更ガード」「保存時の安全性と失敗分類」を重点確認する。

## スコープ

- 閲覧モードから編集モードへの遷移と初期ロード
- 対象書式（本文、見出し1-3、太字、イタリック）の編集と保存
- 保存後の閲覧表示更新
- 未保存変更あり時の離脱ガード（別ファイル選択、編集終了、ブラウザ離脱）
- 未対応要素検知時の警告と保存継続/キャンセル
- 保存対象検証（rootDir 配下、`.md`、include/exclude、パストラバーサル）
- 保存失敗 3 分類（invalid-target / unwritable-target / transient-io）の利用者通知
- 改行コードと末尾改行維持
- 1MB 程度文書での実用性能（p95 3 秒以内）

## スコープ外

- 外部変更の競合検知（Story 011 で扱う）
- 版情報照合、競合時保存拒否
- 下線など対象外書式の編集保証
- ページ再読込後の編集中断データ復元

## テスト観点

1. 基本価値観点

- 編集開始、編集、保存、再閲覧の一連操作が成立するか。

2. 書式意味保持観点

- 見出し1-3、太字、イタリックが Markdown 意味として保存されるか。

3. 安全性観点

- 未保存変更の離脱ガードが 3 トリガーで確実に動作するか。

4. 保存前警告観点

- 未対応要素がある場合に警告が表示され、継続/キャンセルを選べるか。

5. 保存対象検証観点

- 保存先制約違反時に誤保存せず、理由を提示できるか。

6. 障害通知観点

- 失敗 3 分類で理由と再試行可否が利用者へ提示されるか。

7. 形式維持観点

- 改行コードと末尾改行有無を維持できるか。

8. 性能観点

- 1MB 文書で編集開始から保存結果表示まで p95 3 秒以内を満たすか。

## テスト種別設計

- シナリオ手動テスト
  - UI 操作で利用者価値を確認する。
- 境界手動テスト
  - 保存 API を直接呼び出し、保存対象検証と失敗分類を再現する。
- 観測型手動テスト
  - 操作時間、表示メッセージ、保存後ファイル実体を確認する。

## エントリー条件

- `npm install` と `npm run build` が成功している。
- `node dist/cli/index.js serve` が起動し、閲覧画面へアクセスできる。
- Story 010 用の検証文書を準備済み。
- 保存 API に対してローカル環境から `curl` 実行が可能。

## 終了条件

- P1 リリース判定に必要なケース（TC-010-01 から TC-010-08）が完了している。
- TC-010-06 で誤保存 0 件を確認している。
- TC-010-07 で 3 分類の観測結果が記録されている。
- TC-010-08 でファイル形式破壊（改行形式/末尾改行破壊）がない。
- TC-010-09 が未達の場合、性能リスクとして残課題に明示されている。
- TC-010-10（SC-001）はユーザビリティ評価として別枠管理し、未実施時は未評価として記録する。
- 重大不具合（保存不能、誤保存、無警告離脱）が未解決で残っていない。

## リスクと緩和策

- リスク: テストデータ不足で未対応要素や失敗分類を再現できない。
- 緩和策: ケース内で再現用 Markdown と API リクエスト例を明示する。

- リスク: 環境差で I/O 一時失敗を再現しにくい。
- 緩和策: ファイルロックや読み取り専用化など代替再現方法を許容し、再現手段を実測欄に記録する。

- リスク: 性能測定のばらつきが大きい。
- 緩和策: 同一端末で連続 20 回測定し、外れ値の理由を補足欄へ残す。
