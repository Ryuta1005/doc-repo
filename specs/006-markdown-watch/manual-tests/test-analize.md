# 手動テスト分析・設計（Story 006）

## 目的

Story 006 の受け入れ条件と `spec.md` の FR/SC を、人間の操作で end-to-end に検証するための手動テスト設計を定義する。

## 対象範囲

- `doc-repo serve` 実行中の Markdown 変更監視
- `add` / `change` / `unlink` 検知
- 300ms debounce と pending 再実行制御（同時実行なし）
- SSE によるブラウザ自動更新（成功時のみ `reload`）
- 失敗時の no-reload 方針
- `file://` で開いた場合に SSE 接続を行わないこと
- 監視中の進行状況ログ
- `SIGINT` / `SIGTERM` 時の順序付きクリーンアップ

## 非対象

- ブラウザ上での Markdown 編集
- 外部公開サーバー運用（TLS、認証、アクセス制御）
- include/exclude の高度な UI 設定（設定仕様拡張）

## テスト観点

1. 監視検知観点

- `change` で再生成が起動すること
- `add` / `unlink` でも再生成が起動すること
- 非対象ファイル変更で再生成しないこと

2. 更新反映観点

- 再生成成功時のみブラウザが自動更新すること
- 再生成失敗時はブラウザが更新されないこと

3. 再生成制御観点

- 最後の変更から 300ms 後に再生成されること
- 再生成中の追加変更が pending で保持され、完了後に 1 回再実行されること
- 同時に複数再生成が走らないこと

4. 可観測性観点

- 変更検知、再生成開始、成功/失敗、reload 配信の順序が分かること
- エラーが標準エラー出力へ表示されること

5. ライフサイクル観点

- SSE 接続が開始時に登録され、切断時に削除されること
- 停止時に watcher → SSE 接続 → HTTP サーバーの順で停止すること

## テストタイプ設計

- E2E 手動テスト: 主要ユーザージャーニー（US1/US2/US3）
- 異常系手動テスト: 再生成失敗、除外対象、終了制御
- 視覚確認テスト: ブラウザ自動更新/no-reload 確認
- ログ確認テスト: 進行状況の順序確認

## 入口条件

- `npm install` 済み
- `npm run build` 成功済み
- 監視対象に `.md` が 1 件以上ある
- 既存の `doc-repo serve` が停止済み
- ブラウザで表示 URL へアクセス可能

## 終了条件

- 全手動ケースを実行済み
- Fail が 0 件、または Fail について再現手順と原因を記録済み
- FR-001〜FR-013（FR-011 を除く）と SC-001〜SC-005 の全項目に最低 1 件のケース紐付けあり

## リスクと補完策

- リスク: ファイル保存タイミング差で debounce 観測が揺れる
- 補完策: 連続保存手順とログ確認ポイントを固定し、実測値を記録する

- リスク: 失敗注入方法が環境で異なる
- 補完策: テンプレート退避など再現可能な失敗注入を手順化する

- リスク: `file://` 閲覧はブラウザ実装差がある
- 補完策: 少なくとも主要 1 ブラウザで、コンソールエラーなく閲覧できることを確認する

## 実施ログの記録方針

各ケースで以下を記録する。

- 実施日時
- 実施者
- 実行コマンド
- 標準出力/標準エラー要点
- 判定（Pass/Fail/Blocked）
- 補足（再現条件、改善メモ）
